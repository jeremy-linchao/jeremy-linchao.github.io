一个跨域options方法的问题

发表于 2017-04-07   |   分类于 javascript

同事遇到的一个问题，记录一下： 用fetch去跨域时，post使用json，到一个需要cookie的服务提交，报302

查了option的文档 https://www.w3.org/TR/cors/#preflight-request， 当用options的方法时，会exclude掉如下信息

  • Exclude the author request headers.

  • Exclude user credentials.

  • Exclude the request entity body.

    因此，fetch的时候即使你加入了include credientials也没用，解决的办法是让自己的提交是简单请求，不需要预检，是否需要预检如下，

当浏览器的请求方式是HEAD、GET或者POST，并且HTTP的头信息中不会超出以下字段:

  • Accept
  • Accept-Language
  • Content-Language
  • Last-Event-ID
  • Content-Type：只限于三个值application/x-www-form-urlencoded、multipart/form-data、text/plain

时，浏览器会将该请求定义为简单请求，否则就是预检请求。预检请求会在正式通信之前，增加一次HTTP查询请求。浏览器先询问服务器，当前网页所在的域名是否在服务器的许可名单之中，以及可以使用哪些HTTP动词和头信息字段。只有得到肯定答复，浏览器才会发出正式的XMLHttpRequest请求，否则就报错。预检请求的发送请求：

数据库系统(mooc)

发表于 2017-04-03   |   分类于 database

  1. 关系与表的关系： 关系是关系模型里一个数学概念，而表是DBMS实现时用的一个称呼，两者在大多数情况下都是相同的，但关系里不能有重复的record，表里可以有

  2. 如何描述一个关系: 1) 先确定域，域的所有取值是能够确定下来的，同时有多少个域也是能确定下来的

    2）使用域作笛卡尔乘积，从中挑选一些有意义的record的集合作为一个关系

  3. 完整性约束： 1） 实体完整性： 主键不能为空 2）参照完整性： 外键的值的取值为null或者为参照表的主键中的取值 3） 自定义完整性： 用户自己定义的约束，比如字符串长度，数字长度

  4. 候选键（能唯一标识所有用户的属性组） 主键： 从候选键中选出一个属性组 外键： 在某一个关系下不是候选键，但在另一个关系下是主键，用来连接关系​

  5. 关系代数的基本动作： 并， 差， 乘积， 选择， 投影

  6. 并相容性： 两个关系的属性的数量以及每个属性的取值范围必须相同。 并，交， 差运算需满足并相容性

  7. theta-join, 等值连接: 特殊的theta-join

  8. Natual-join: 特殊的等值连接

  9. 外连接： 为什么需要外链接？内连接由于是需要属性相同并且值也相同，因此会丢失一部分信息。

    定义：连接时如果在另外一张表中找不到对应的元祖，则以空值作为代替连接

  10. DDL: CREATE, ALTER, DROP

    DML: INSERT, DELETE, UPDATE, SELECT

    DCL: GRANT, REVOKE

  11. 批量插入 INSERT INTO Table(col1, col2) 子查询

  12. 删除： DELETE FROM 表名 [WHERE 表达式]

  13. 更新： UPDATE 表名 SET 列名 = 表达式|（子查询）[WHERE 条件表达式]

  14. 子查询： 出现在Where子句中的Select语句， IN 与 NOT IN； theta some | all

  15. E-R模型： Enitity-Relation Model，四个要素：实体、属性、联系、关键字

  16. 一对多和多对多关系的区别: 一对多的联系可以通过外键来保存，对多对的联系只能单独再建一个联系表

  17. 一对多联系如何转换： 将单方参与实体的关键字作为多方参与实体对应关系的属性

docker学习

发表于 2017-03-27   |   分类于 docker

一. 为什么docker？

抛开微服务，抛开持续集成，docker对我吸引最大的一点就是可以保持本地开发环境与线上开发环境的一致性，前端遇到此类的问题相对较少些，因为完全可以本地编译成bundle然后上传，顶多只要保持nginx的环境一致即可，这相对简单。但是如果涉及到服务端开发，那依赖的环境就太多了，数据库，node版本等等，如果能保持本地和线上环境的一致，一次build，到处运行，这个对于开发和运维的吸引力实在太大了！！！

https://yeasy.gitbooks.io/docker_practice

这个电子书不错

  1. docker其实也是一个client和server的架构，docker daemon就是核心的后台进程，它就是在后台启动了一个API server，负责由docker client发送来的请求， 而client的版本很多，只要遵循docker api即可

  2. docker容器的本质是宿主机上的进程，通过namespace实现资源隔离，通过cgroup实现资源限制，通过copy-on-write实现高效的文本操作

二. docker machine 和 docker engine的区别

  1. docker engine是我们常说的docker，c-s架构，包括docker daemon和client

  2. Docker Machine is a tool for provisioning and managing your Dockerized hosts

三. Docker hub加速

  1. 如果用的是最新的docker for mac, 只需要在docker的preference里的daemon选项卡里增加个registry mirror，至于地址可以用阿里云或蜂巢的镜像加速功能

四. registry和repository的区别

  1. registry是注册服务器，repository是仓库，一个服务器上可以有多个仓库，仓库可以理解为某个目录或某个项目，dl.dockerpool.com/ubuntu 对于这个地址来说，dl.dockerpool.com是注册服务器，ubuntu是仓库名

五. Docker的build

  1. docker build [选项] <上下文路径/URL/->

当我们进行镜像构建的时候，并非所有定制都会通过 RUN 指令完成，经常会需要将一些本地文件复制进镜像，比如通过 COPY 指令、ADD 指令等。而 docker build 命令构建镜像，其实并非在本地构建，而是在服务端，也就是 Docker 引擎中构建的。那么在这种客户端/服务端的架构中，如何才能让服务端获得本地文件呢？

这就引入了上下文的概念。当构建的时候，用户会指定构建镜像上下文的路径，docker build 命令得知这个路径后，会将路径下的所有内容打包，然后上传给 Docker 引擎。这样 Docker 引擎收到这个上下文包后，展开就会获得构建镜像所需的一切文件。
