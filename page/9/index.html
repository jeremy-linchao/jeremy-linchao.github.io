API设计

发表于 2016-12-14   |   分类于 concept

  • 所有API请求都使用https

  • 在response的accept header上加一个版本号，比如 version = 3

  • 在所有的response里支持ETags，这样可以允许用户来缓存资源

  • 在每个header中包含一个request-id，由uuid构成，目的是为了方便调试和追踪

  • 大的response应该用Range头来指定如何获取这些数据
    ​

    Request

    1. post，put等这些请求的body中只接受序列化的json格式，不要用类似form-endoded的方式，这样可以保持请求和返回的一致
    2. 尽量用复数（加s），除非那个资源是个单实例
    3. 对于每个独立资源，尽量不使用任何special actions，如果需要action，也放在单独的action下 /resources/:resource/actions/:action
      1. 使用小写字母，对于路径名使用这个划线 “-“，对于属性名，使用下划线_
      2. 路径长度最小化

        Response

  1. 返回合适的状态码

    200： GET，POST，DELETE，PATCH同步操作顺利完成

    201：POST，PUT异步创建资源成功

    202：POST，PUT，DELETE异步操作

    206：GET操作成功，但只返回一部分数据

    401 Unauthorized： 鉴权失败

    403 Forbidden：对于某一个资源没有特别的权限

    422：无效参数

    429: 请求次数太频繁，做速率限制

    500：内部服务器错误

  2. 对于每一个资源提供一个UUID

  3. 对于每个资源提供时间戳

  4. 使用UTC times ISO8601

  5. 使用嵌入对象做外键引用

对前端的一些感悟

发表于 2016-12-14   |   分类于 thinking

​ 做前端差不多有一个多月了，之间也看过听到了大家对前端的一些感悟，比如那篇转发很多的《2016年学javascript》是一个什么样的体验，在我第一次看的时候我也觉得“哇靠，前端都在折腾些啥啊“，但当我慢慢地看深入一些，发现事情并不是这样。

​ 要理解前端的这些变化，必须要理解单页应用。在很早期的时候，web的前端主要是一个负责html解析，样式和动画的渲染，每一次用户点击一个链接或者一个button，请求都是要发到服务器端，服务器或直接返回静态页面，或者从数据库拉取数据然后生成html返回浏览器，这里面的页面渲染和路由都是在服务端做的，但仔细一想会觉得这个其实不合理，浏览器大多数情况下只要更新数据，为什么每次都要重新渲染个html页面过来呢，一来会增加传输带宽，二来呢主要是影响用户体验，毕竟在网络情况不是太好的时候这是个很大问题。

​ 这时候回想传统的CS架构，为什么觉得CS架构用户体验会好，除了开发语言C/C++直接编译成二进制的优势，很大程度上是因为CS架构的很多UI操作都是在本地完成，与服务器通信也只需要通过tcp获取需要的数据，所以我觉得单页应用并没有什么太新的概念，他只不过是回归了本来应该的样子，浏览器在第一次加载的时候从服务器拉取了下这个应用的代码。为什么以前没有，我想一方面是浏览器对于js的程度越来越好，另外就是大家对用户体验越来越看重。

​ 如果理解了单页应用，那随后几个概念其实很好理解，也能理解为什么会有。首先是webpack，webpack是一个解决前端依赖和打包的工具，因为一旦路由和逻辑放到了前端，前端的代码量会急剧上升，很多模块间会需要通过做多个导入，但是这些模块又会有相互依赖，webpack的一个作用就是来解决js加载的顺序问题，另外js代码的压缩及混淆工作也是一个非常重要的事情，不然那么多js代码，首次加载会非常之慢。

​ Babel是做了另外一方面的事情，是关于语法，ES6的推出确实是一件非常振奋人心的事，里面有很多我非常喜欢的特性，比如let的块级作用，比如解构赋值，比如promise对象，但由于浏览器的支持总是分阶段的，所以为了最大的兼容性，还是需要ES5的代码，Babel就是干了这么一件事，把ES6以及之后的代码转换成ES5的

Git reset 和 git checkout 的区别

发表于 2016-12-09   |   分类于 git

  1. git chekout 是用来移动HEAD指向不同的分支，git reset是在同一个分支内移动HEAD头

  2. git reset - - soft HEAD~ 只移动HEAD到那个commit，注意此时staged区和工作目录并没有被重置

    git reset - - mixed HEAD~ 移动HEAD到那个commit，staged被重置到那个commit，但工作目录没有

    git reset - -hard HEAD~ 移动HEAD到那个commit，并且同时重置staged区域以及工作目录，这个因为会覆盖工作目录会有一点风险性，如果你当前修改的东西没有commit然后hard的话当前的找不到，当然如果是简单想回去看看，那是没有问题的

  3. git reset: 可以作用于commit级别，但是它的主要用途是撤销某个提交
  4. git checkout: 也可以作用于commit级别，它的用途是查看之前某个version，在查看github源代码的时候特别有用，因为不会覆盖
